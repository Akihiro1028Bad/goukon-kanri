# Issue #25: スマホ表示時のデザインの修正

## 問題分析

### 現状の動作
スマホで開いた時に、以下の箇所で見づらい・操作しづらい問題が発生しています。

### 問題の原因（コード箇所を特定）

#### 1. テーブルの横スクロール問題
**問題箇所:**
- `src/components/dashboard/monthly-summary-table.tsx` (112行目)
- `src/components/events/event-table.tsx` (152行目)
- `src/components/participants/participant-table.tsx` (278行目)
- `src/components/schedule/schedule-table.tsx` (179行目)
- `src/components/reports/report-table.tsx` (115行目)

**問題内容:**
- すべてのテーブルに `overflow-x-auto` が設定されているが、カラム数が多い（特にダッシュボードは13カラム、参加者テーブルは11カラム、スケジュールテーブルは13カラム）
- スマホの狭い画面では、`whitespace-nowrap` により各セルが改行されないため、テーブル全体が非常に横長になる
- 横スクロールの範囲が広すぎて、一度に全体を把握できず、重要な情報を見落とす可能性がある
- 特に、参加者テーブルではアクションボタン（編集・削除）が右端にあり、到達するまでのスクロールが必要

#### 2. ページヘッダーの折り返し問題
**問題箇所:**
- `src/app/page.tsx` (21-24行目)
- `src/app/events/page.tsx` (36-44行目)

**問題内容:**
```tsx
<div className="flex items-center justify-between">
  <h1 className="text-2xl font-bold">ダッシュボード</h1>
  <YearSelector currentYear={year} />
</div>
```
- `flex` + `justify-between` で左右に配置しているが、スマホでは画面幅が狭いため、タイトルとボタンが重なる、または不自然な配置になる
- `text-2xl` (24px) のタイトルとセレクタが1行に収まらない可能性がある

#### 3. イベント詳細ページの情報密度問題
**問題箇所:**
- `src/components/events/event-detail.tsx` (141行目, 201行目, 228行目)

**問題内容:**
```tsx
<dl className="grid grid-cols-1 md:grid-cols-2 gap-4">  // 基本情報
<dl className="grid grid-cols-2 md:grid-cols-4 gap-4">  // 定員・参加費
<dl className="grid grid-cols-2 md:grid-cols-3 lg:grid-cols-4 gap-4">  // 収支サマリー
```
- 基本情報: `grid-cols-1` でモバイルは1カラムだが、縦長になりすぎる
- 定員・参加費: モバイルでも `grid-cols-2` だが、ラベルが長い場合（「男性参加費」など）テキストが折り返され見づらい
- 収支サマリー: モバイルで `grid-cols-2` だが、12項目あるため縦に長くなりすぎる
- `text-sm font-medium text-muted-foreground` のラベルと値が近接しており、視認性が低い

#### 4. イベントフォームの入力フィールド問題
**問題箇所:**
- `src/components/events/event-form.tsx` (101行目以降)

**問題内容:**
```tsx
<div className="grid grid-cols-1 md:grid-cols-2 gap-4">  // 日付・開始時刻
<div className="grid grid-cols-2 gap-4">  // 男女定員、男女参加費
<div className="grid grid-cols-1 md:grid-cols-2 gap-4">  // テーマ・対象職業
```
- 定員と参加費は常に `grid-cols-2` だが、スマホでは入力フィールドが狭くなりすぎる
- 数値入力フィールドが小さく、タップしづらい

#### 5. フィルター要素の折り返し問題
**問題箇所:**
- `src/components/events/event-filters.tsx` (38行目)

**問題内容:**
```tsx
<div className="flex flex-wrap gap-4">
```
- `flex-wrap` で折り返すが、セレクトボックスのサイズが固定（`w-[120px]`, `w-[100px]`, `w-[130px]`）
- スマホでは折り返しが多く発生し、縦方向のスペースを過度に消費する
- 年度・月・状態の3つのフィルタが1行に収まらない

#### 6. 参加者テーブルのツールバー問題
**問題箇所:**
- `src/components/participants/participant-table.tsx` (246行目)

**問題内容:**
```tsx
<div className="flex flex-wrap items-center gap-4">
  <Input className="max-w-xs" />
  <div className="flex items-center gap-2">...</div>
  <BulkPaymentDialog>...</BulkPaymentDialog>
</div>
```
- 検索ボックス、スイッチ、一括決済ボタンが横並びで、スマホでは複数行に折り返す
- `max-w-xs` (384px) の検索ボックスがスマホの画面幅を超える場合がある

#### 7. レイアウトのパディング問題
**問題箇所:**
- `src/app/layout.tsx` (21行目)

**問題内容:**
```tsx
<main className="min-w-0 flex-1 p-4 pt-16 md:p-6 md:pt-6">
```
- モバイルでは上部に固定ヘッダー（ハンバーガーメニュー）があるため `pt-16` (64px) が設定されているが、コンテンツ領域が狭くなる
- `p-4` (16px) のパディングで左右のスペースが確保されているが、テーブルなどの横幅の大きい要素では、さらに狭く感じる

#### 8. イベント詳細ページのヘッダー問題
**問題箇所:**
- `src/components/events/event-detail.tsx` (113-131行目)

**問題内容:**
```tsx
<div className="flex items-center justify-between">
  <div>
    <h1 className="text-2xl font-bold">イベント {event.eventId}</h1>
    <Badge variant={statusVariant[event.status]} className="mt-2">
      {EVENT_STATUS_LABELS[event.status]}
    </Badge>
  </div>
  <div className="flex gap-2">
    <Link className={buttonVariants({ variant: "outline" })}>編集</Link>
    <Button variant="destructive">削除</Button>
  </div>
</div>
```
- `justify-between` でタイトル部分とボタン群を両端配置しているが、スマホではタイトルが長い（イベントIDが長い）場合、ボタンが右端に押し付けられる
- 編集・削除ボタンがタイトルと同じ高さにあるため、縦のスペースを有効活用できていない

## 修正方針

### 変更対象ファイルと変更内容

#### 1. テーブルコンポーネントのレスポンシブ対応

**対象ファイル:**
- `src/components/dashboard/monthly-summary-table.tsx`
- `src/components/events/event-table.tsx`
- `src/components/participants/participant-table.tsx`
- `src/components/schedule/schedule-table.tsx`
- `src/components/reports/report-table.tsx`

**変更内容:**
- モバイル表示時に、重要なカラムのみを表示し、残りは非表示にする
- `hidden md:table-cell` などのTailwindユーティリティクラスを使用して、カラムごとに表示・非表示を制御
- 各テーブルごとに、以下の方針で優先度を設定:
  - **月次サマリーテーブル**: 月、件数、見込み収入、見込み利益のみモバイル表示
  - **イベントテーブル**: イベントID、日付、会場名、状態のみモバイル表示
  - **参加者テーブル**: 氏名、性別、決済状態、アクションボタンのみモバイル表示
  - **スケジュールテーブル**: イベントID、日付、会場名、男女残枠、LINEボタンのみモバイル表示
  - **レポートテーブル**: イベントID、日付、見込み収入、見込み利益のみモバイル表示

#### 2. ページヘッダーのレスポンシブ対応

**対象ファイル:**
- `src/app/page.tsx`
- `src/app/events/page.tsx`

**変更内容:**
- モバイル時は縦積みレイアウトに変更
- `flex items-center justify-between` を `flex flex-col md:flex-row items-start md:items-center gap-4 md:justify-between` に変更
- タイトルとアクション要素が縦に並ぶことで、十分なタップ領域を確保

#### 3. イベント詳細ページのグリッド調整

**対象ファイル:**
- `src/components/events/event-detail.tsx`

**変更内容:**
- 基本情報: `grid-cols-1 md:grid-cols-2` のままで問題なし
- 定員・参加費: `grid-cols-1 sm:grid-cols-2 md:grid-cols-4` に変更し、小さいスマホでは1カラム表示
- 収支サマリー: `grid-cols-1 sm:grid-cols-2 lg:grid-cols-4` に変更し、項目数が多いため段階的にカラム数を増やす
- イベント詳細ページヘッダー: `flex flex-col md:flex-row items-start md:items-center gap-4 md:justify-between` に変更し、モバイルでは縦積み

#### 4. イベントフォームのグリッド調整

**対象ファイル:**
- `src/components/events/event-form.tsx`

**変更内容:**
- 定員フィールド: `grid-cols-1 sm:grid-cols-2` に変更
- 参加費フィールド: `grid-cols-1 sm:grid-cols-2` に変更
- モバイルでは各入力フィールドを縦積みにすることで、タップ領域と視認性を向上

#### 5. フィルターのレスポンシブ対応

**対象ファイル:**
- `src/components/events/event-filters.tsx`

**変更内容:**
- セレクトボックスの幅を固定値から `w-full sm:w-[120px]` などに変更し、モバイルでは全幅を使用
- グリッドレイアウトに変更: `grid grid-cols-1 sm:grid-cols-3 gap-3`
- モバイルでは各フィルタが縦に並び、タップしやすくなる

#### 6. 参加者テーブルツールバーの調整

**対象ファイル:**
- `src/components/participants/participant-table.tsx`

**変更内容:**
- ツールバー全体を `flex flex-col sm:flex-row flex-wrap items-stretch sm:items-center gap-3` に変更
- 検索ボックスの `max-w-xs` を `w-full sm:max-w-xs` に変更し、モバイルでは全幅を使用
- 一括決済ボタンも `w-full sm:w-auto` に変更し、モバイルでは全幅ボタンに

#### 7. レイアウトのパディング調整（検討のみ）

**対象ファイル:**
- `src/app/layout.tsx`

**変更内容:**
- 現状の `p-4 pt-16 md:p-6 md:pt-6` は妥当な設定と判断
- 必要に応じて、テーブルコンテナに `-mx-4 md:mx-0` を追加し、モバイルで左右のパディングを打ち消すことも検討可能（ただし、今回は不採用）

#### 8. タスク進捗サマリーの調整

**対象ファイル:**
- `src/components/events/event-detail.tsx`

**変更内容:**
- タスク進捗の `grid-cols-1 md:grid-cols-3` はそのままだが、モバイルで縦積みになりすぎるため `grid-cols-1` に変更

### 選択しなかった代替案とその理由

#### 代替案1: カード型レイアウトへの全面変更
**理由:**
- テーブルをカード型（各行を1つのカードとして表示）に変更する方法もあるが、実装コストが高い
- データの比較がしづらくなる（特に数値データが多い月次サマリーやレポート）
- 既存のテーブルコンポーネントの構造を大きく変更する必要があり、破壊的変更となる

#### 代替案2: テーブルの完全な再設計（固定カラム + 横スクロール）
**理由:**
- 左端のカラム（例: イベントID）を固定し、残りを横スクロールする方法もあるが、CSSの `position: sticky` の実装が複雑
- ブラウザ互換性の問題やz-indexの管理が煩雑になる
- 今回の要件（見づらい・操作しづらいの改善）に対して、オーバーエンジニアリングと判断

#### 代替案3: モバイル専用ページの作成
**理由:**
- PC版とモバイル版で完全に別のコンポーネントを用意する方法もあるが、保守性が低下
- コードの重複が発生し、バグの温床となる
- レスポンシブデザインのTailwindの仕組みで十分対応可能

#### 代替案4: フィルターのドロップダウン化
**理由:**
- フィルター全体を1つのドロップダウンメニューに格納する方法もあるが、操作性が低下
- ユーザーは複数のフィルタを切り替えながら使用するため、常に表示されている方が望ましい

## 補足事項

- **Tailwindのブレークポイント:**
  - `sm`: 640px以上
  - `md`: 768px以上
  - `lg`: 1024px以上
  - `xl`: 1280px以上

- **テーブルのカラム非表示について:**
  - `hidden md:table-cell` は、モバイル（0-767px）で非表示、タブレット以上（768px以上）で表示
  - `table-cell` はテーブルセルとしての表示を維持するために使用

- **パフォーマンスへの影響:**
  - `hidden` クラスは `display: none` を適用するため、要素はDOMに存在するがレンダリングされない
  - 大量のデータがある場合でも、パフォーマンスへの影響は最小限

- **アクセシビリティ:**
  - モバイルで非表示にしたカラムの情報は、各行のリンク（イベントIDなど）をタップして詳細ページで確認可能
  - スクリーンリーダーは `hidden` 要素を読み上げないため、代替手段（詳細ページへのリンク）を提供することが重要

- **将来的な拡張:**
  - カード型レイアウトへの移行が必要になった場合、`@media (max-width: 767px)` で `table` 要素を `div` に置き換え、カードスタイルを適用する方法も検討可能
  - ただし、現時点ではカラム非表示で十分な改善が見込まれるため、将来的な検討事項とする
